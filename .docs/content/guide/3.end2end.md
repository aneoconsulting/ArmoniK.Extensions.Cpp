# Build and configure your own client/worker test

You can also develop a custom client to submit jobs to ArmoniK and invoke the desired services embedded in the worker shared library.

## Client test

The SessionService class in the `ArmoniK::Sdk::Client` namespace facilitates client creation. This class includes methods for sending tasks to ArmoniK and retrieving the results, specifically the `Submit` and `WaitResults` methods.

### Configure client

The constructor of the SessionService class takes as arguments a `properties` and a `logger` objects. The `properties` object as defined in `ArmoniK::Sdk::Common` namespace contain the configuration and a session task options objects.

- The configuration object is loaded from a JSON file or from environment variables that define the following:
  - Endpoint: The control plane address to which clients connect.
  - mTLS: enable or disable mTLS.
  - caCert: The Server certificate file to validate mTLS.
  - clientCert: The client certificate fil in a pem format.
  - clientKey: The client key file in a pem format.
  - sslValidation: Disable the ssl strong validation of ssl certificate (default: enable => true).

The environment variables override the configuration in the JSON files.
- The session task options contain information required by a session such as the following:
  - MaxRetries (default value is set to 3)
  - Priority (default value is set to 2)
  - AppName the name of the worker shared library (legacy path) or a descriptive name (convention path)
  - AppVersion the version of the library (can be empty)
  - AppNamespace the service namespace
  - AppService the service name
  - For the convention path: `DynamicLibrary` configuration encoded via `SetDynamicLibrary` (see below)

### Setup your own client

To set up a `SessionService`, specify `properties` and a `logger` as described above.

#### Invocation handler

Before submitting tasks, implement `IServiceInvocationHandler`. Override `HandleResponse` (three-parameter version) and `HandleError`:

```cpp
class MyHandler : public ArmoniK::Sdk::Client::IServiceInvocationHandler {
public:
  void HandleResponse(const std::string &result_payload,
                      const std::string &taskId,
                      const std::string &result_id) override {
    // result_payload: the worker's return value
    // result_id: blob ID — pass to BlobDefinition::FromBlobId for task chaining
  }

  void HandleError(const std::exception &e, const std::string &taskId) override {
    // handle task failure
  }
};
```

The two-parameter overload `HandleResponse(result_payload, taskId)` is still accepted for backward compatibility but is deprecated; override the three-parameter version for new code.

#### Submitting tasks — convention path (recommended)

The convention path uses `TaskDefinition` with named inputs. The SDK uploads raw data automatically on submission; no manual blob pre-allocation is required.

Configure a `DynamicLibrary` and attach it to the task options with `SetDynamicLibrary`:

```cpp
ArmoniK::Sdk::Common::DynamicLibrary lib;
lib.library_path = "/data/libMyWorker.so";  // path on the worker filesystem
lib.symbol = "square";                      // method name forwarded to call()

ArmoniK::Sdk::Common::TaskOptions opts("libMyWorker.so", "1.0.0",
                                       "MyNamespace", "MyService", partitionId);
opts.SetDynamicLibrary(lib);
```

If the worker library is not present on the worker's local filesystem, upload it first with `UploadLibrary` and the blob ID is set automatically:

```cpp
service.UploadLibrary("/path/to/libMyWorker.so", lib);
// lib.library_blob_id is now set; the worker fetches the .so from blob storage at runtime
opts.SetDynamicLibrary(lib);
```

Then submit tasks using `TaskDefinition`. Inputs are named blobs:

```cpp
ArmoniK::Sdk::Client::SessionService service(properties, logger);
auto handler = std::make_shared<MyHandler>();

service.Submit(
    {ArmoniK::Sdk::Common::TaskDefinition(
        "square",
        {{"x", ArmoniK::Sdk::Common::BlobDefinition::FromData("7")}})},
    handler, opts);

service.WaitResults();
```

`BlobDefinition::FromData(data)` uploads raw bytes. `BlobDefinition::FromBlobId(id)` references an already-uploaded blob without re-uploading.

#### Submitting tasks — legacy path

The legacy path uses `TaskPayload` with a pre-serialized binary payload. It remains functional but `TaskPayload` is deprecated; new code should prefer `TaskDefinition`.

```cpp
service.Submit(
    {ArmoniK::Sdk::Common::TaskPayload("methodName", serialized_args)},
    handler);
```

#### Task chaining with `result_id`

`HandleResponse` provides `result_id`, the blob ID of the completed task's output. Pass it to `BlobDefinition::FromBlobId` to use that result as an input for a downstream task — no re-upload needed:

```cpp
void HandleResponse(const std::string &payload,
                    const std::string &taskId,
                    const std::string &result_id) override {
  this->last_result_id = result_id;
}
```

```cpp
// After task A completes:
service.Submit(
    {ArmoniK::Sdk::Common::TaskDefinition(
        "add",
        {{"a", ArmoniK::Sdk::Common::BlobDefinition::FromBlobId(handler_a->last_result_id)},
         {"b", ArmoniK::Sdk::Common::BlobDefinition::FromBlobId(handler_b->last_result_id)}})},
    handler_c, opts);
```

#### Waiting for results

`WaitResults` retrieves results of submitted tasks. If called with no arguments it waits for all tasks submitted with this service:

```cpp
service.WaitResults();  // wait for all
```

To wait for a specific subset, or to control the wait behaviour:

- `task_ids`: set of task IDs to wait on; if empty, waits for all submitted tasks.
- `waitBehavior`: `All` (default) or `Any`; add `BreakOnError` to exit as soon as a result is aborted.
- `waitOptions`: controls the polling interval.

## Worker

The ArmoniK dynamic worker deployed with the infrastructure is responsible for receiving tasks from clients and executing them. It supports two execution paths:

- **Convention path** (recommended): tasks submitted with `TaskDefinition` carry a JSON payload with named inputs; the worker receives them as a `std::map<std::string, std::string>`. Compatible with any ArmoniK SDK (C++, Java, C#).
- **Legacy path**: tasks submitted with `TaskPayload` carry a binary payload; the worker receives the raw bytes.

In both cases the implementation of the methods lives in a worker-shared library as a `ServiceBase` object. The following sections explain how to build this library.

### Worker service implementation

To create an ArmoniK Worker Service, a shared library exposing 5 mandatory symbols must be created (cf ArmoniK SDK Interface). This library must then be deployed either by copying it to the shared storage, which depends on your deployment:

- For a deployment using a `HostPath` type (the default type for a local host deployment), the directory is:
```
kubectl -n armonik get secret -o template={{.data.host_path}} shared-storage | base64 -d
```
- For a `s3` storage type, the bucket url can be recovered by
```
kubectl -n armonik get secret -o template={{.data.service_url}} shared-storage | base64 -d
```

**Legacy path:** `application_name` and `application_version` task options determine the library filename (`application_name.application_version` or `application_name`). Copy the `.so` to shared storage as described above.

**Convention path:** set `DynamicLibrary.library_path` to the `.so` path on the worker filesystem, or use `SessionService::UploadLibrary` to upload the `.so` to ArmoniK blob storage — in that case the worker downloads the library at runtime and `library_path` is optional. The library is identified via `SetDynamicLibrary` in the task options rather than by filename convention.

The calls to the symbols and the expected lifetimes of the service and sessions within a worker pod are described in the flowchart presented in the appendix section.

### Setup worker shared lib

Link your code to the `ArmoniK.SDK.Worker` static library. It provides default implementations of the mandatory C symbols (`armonik_enter_session`, `armonik_call`, `armonik_leave_session`, `armonik_destroy_service`). The library targets C++14 or later.

Derive a service class from `ArmoniK::Sdk::Worker::ServiceBase` and implement `armonik_create_service` (not provided by `ServiceBase`). Examples: [Echo service](https://github.com/aneoconsulting/ArmoniK.Extensions.Cpp/blob/main/ArmoniK.SDK.Worker.Test/include/EchoService.h), [Addition service](https://github.com/aneoconsulting/ArmoniK.Extensions.Cpp/blob/main/ArmoniK.SDK.Worker.Test/include/AdditionService.h), [ServiceDispatch.cpp](https://github.com/aneoconsulting/ArmoniK.Extensions.Cpp/blob/main/ArmoniK.SDK.Worker.Test/src/ServiceDispatch.cpp).

#### Convention-path worker (recommended for cross-SDK interoperability)

Override the map-based `call()` overload. The SDK parses the convention JSON payload automatically and delivers named inputs as a `std::map<std::string, std::string>` — no JSON parsing required in the worker:

```cpp
#include <armonik/sdk/worker/ServiceBase.h>

class MyService : public ArmoniK::Sdk::Worker::ServiceBase {
public:
  std::string call(void *session_ctx, const std::string &name,
                   const std::map<std::string, std::string> &inputs) override {
    if (name == "square") {
      int x = std::stoi(inputs.at("x"));
      return std::to_string(x * x);
    }
    throw std::runtime_error("unknown method: " + name);
  }
};
```

Register the service in `armonik_create_service`:

```cpp
extern "C" void *armonik_create_service(const char *ns, const char *name) {
  if (std::strcmp(name, "MyService") == 0)
    return new MyService();
  throw std::runtime_error(std::string("unknown service: ") + name);
}
```

No additional CMake dependencies are needed beyond `ArmoniK.SDK.Worker`; nlohmann/json is an internal implementation detail of the SDK, not a direct dependency of worker code.

Convention-path workers are compatible with clients written in any supported ArmoniK SDK (C++, Java, C#) that use the same convention format.

#### Legacy-path worker

Override the string-based `call()` overload to receive the raw serialized payload:

```cpp
std::string call(void *session_ctx, const std::string &name,
                 const std::string &input) override {
  // deserialize `input` using your own format
  return serialized_result;
}
```

Legacy workers use `application_name` / `application_version` task options to identify the library; the library name on the worker filesystem must match `application_name.application_version` or `application_name`.

#### Raw symbols (advanced)

If you prefer not to inherit `ServiceBase`, implement the five mandatory C symbols directly:

- `armonik_create_service` — creates and returns a service context (may return null).
- `armonik_enter_session` — called on session start; returns a session context.
- `armonik_call` — executes a method and sends the result via the provided callback.
- `armonik_leave_session` — frees session context resources.
- `armonik_destroy_service` — frees service context resources.

See the [ArmoniKSDKInterface.h documentation](https://armonikextensionscpp.readthedocs.io/en/latest/content/cpp/index.html#ArmoniKSDKInterface_8h) for the full symbol signatures.
